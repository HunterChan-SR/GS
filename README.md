# 公安局-税务局跨部门隐私求交协议

---

## 一、文档概述

### 1.1 协议目标

针对公安与税务跨部门数据联合查询的业务需求协议。协议实现以下核心目标：

1. 公安方（检索方）可完成目标用户的存在性查询，全程不泄露自身检索的索引信息；
2. 税务方（数据持有方）可完成数据授权匹配，全程不泄露全量索引与业务明文数据；
3. 仅当目标索引匹配成功时，公安方可获取对应单条业务数据的解密权限，严格遵循数据共享“最小必要”原则；

### 1.2 系统角色定义

| 参与方        | 角色定位               | 核心权限与能力                                                                     |
| ------------- | ---------------------- | ---------------------------------------------------------------------------------- |
| A方（公安方） | 检索请求方、协议发起方 | 持有待检索的用户索引数据，仅可获取匹配成功的单条业务数据，无法获取税务方全量数据   |
| B方（税务方） | 数据持有方、密钥管理方 | 持有全量业务数据、系统主密钥，负责数据加密与索引构建，无法获取公安方的检索索引信息 |

### 1.3 术语与符号说明

| 符号/术语          | 完整定义                                                                         |
| ------------------ | -------------------------------------------------------------------------------- |
| $MSK$            | 系统主密钥（Master Secret Key），由税务方本地安全持有，全程不对外泄露            |
| $id$             | 数据唯一索引标识，为每条业务数据的检索主键，全局唯一                             |
| $g$              | 大素数阶乘法循环群$G$的生成元，离散对数问题满足密码学安全要求                  |
| $K_{id}$         | 单条数据的分片对称解密密钥，$K_{id}=g^{\frac{1}{MSK+id}}$                      |
| $\beta_i$        | 税务方构建的二叉树第$i$层节点的索引值                                          |
| $\alpha_i$       | 公安方检索索引在二叉树第$i$层对应的检索索引值                                  |
| $k_i$            | 税务方为二叉树第$i$层生成的随机盲化数                                          |
| $r,R$            | 公安方生成的密码学安全随机数，用于盲化隐私数据，$r\in[0,2^l]$，$l$为安全参数 |
| $msb(x)$         | 取数值$x$的最高有效位（符号位），用于判断数值正负性                            |
| $HE$             | 语义安全的加法同态加密算法，支持密文加法、密文数乘同态运算                       |
| $Enc_{HE}(x,pk)$ | 用同态公钥$pk$对明文$x$执行同态加密                                          |
| $Dec_{HE}(c,sk)$ | 用同态私钥$sk$对密文$c$执行同态解密                                          |
| $\oplus$         | 二进制异或运算                                                                   |

### 1.4 安全模型

本协议基于**半诚实敌手模型**设计：即协议参与方会严格遵守协议执行流程，但会尝试从接收到的交互信息中推断额外的隐私信息；协议可抵御半诚实敌手的隐私窃取攻击，保障双方数据隐私。

## 二、协议离线预处理阶段

本阶段所有操作由税务方（B方）离线完成，不占用在线查询资源，为协议执行完成前置准备。

### 步骤1：全量索引数据排序

生成全量10亿条业务数据对应的唯一索引$id$，对所有索引按升序完成排序，构建有序全量索引库。

### 步骤2：索引分桶与存储

设定分桶模数$M$，对排序后的所有索引$id$执行模$M$运算，按运算结果完成数据分桶；将分桶后的索引与对应业务数据关联，存储至本地安全数据库。

### 步骤3：业务数据加密

对每条业务数据，使用对应分片密钥$K_{id}=g^{\frac{1}{MSK+id}}$执行国密对称加密（SM4）；加密后的业务数据与对应$id$关联存储，明文业务数据与主密钥$MSK$仅在税务方本地留存。

### 步骤4：桶内二叉树索引构建

对每个分桶内的有序索引，构建平衡二叉查找树；二叉树每一层节点的索引值记为$\beta_i$（$i$为二叉树层数），完成全部分桶的二叉树索引构建，实现检索复杂度从$O(n)$到$O(\log n)$的优化。

## 三、协议在线执行核心流程

本阶段为协议在线交互流程，分为**隐私存在性判断**与**解密密钥安全协商**两个核心阶段，双方通过安全信道完成所有数据交互。

### 3.1 阶段一：隐私存在性判断（安全二叉树检索）

本阶段核心目标：在不泄露双方索引明文的前提下，通过安全数值比较协议完成二叉树隐私遍历，判断公安方检索的$id$是否存在于税务方数据库。

**前置条件**：公安方（A方）已确定待检索的目标$id$，基于与税务方一致的分桶规则，确定目标$id$对应的分桶，后续检索仅针对该分桶的二叉树执行。

#### 步骤1：税务方二叉树盲化处理

1. B方为目标分桶二叉树的每一层$i$，生成密码学安全随机数$k_i$；
2. 对每层节点的索引$\beta_i$，计算盲化值$\beta_i - k_i$；
3. 将盲化后的二叉树（仅包含每层$\beta_i - k_i$信息）、对应分桶的加密业务数据，通过安全信道发送至A方。

#### 步骤2：双方逐轮执行安全数值比较协议

A方基于待检索的目标$id$，计算二叉树每层对应的检索索引值$\alpha_i$；双方针对二叉树的每一层，逐轮执行以下安全比较协议，判断$\alpha_i$与$\beta_i$的大小关系，确定二叉树的遍历方向：

- a.  本地预处理计算

  - A方本地计算：$d_A = \alpha_i - (\beta_i - k_i)$
  - B方本地计算：$d_B = -k_i$
- b.  A方盲化处理
  A生成密码学安全随机数$r\in[0,2^l]$（$l$为与数值位数匹配的安全参数，128位），计算$e_A = d_A + r$，将$e_A$发送至B方。
- c.  B方符号位计算与返回
  B方本地计算$e = d_B + e_A = \alpha_i - \beta_i + r$，提取$e$的最高有效位（符号位）$msb(e)$，将$msb(e)$发送至A方。
- d.  A方解盲与大小判断
  A方本地计算$d = msb(e) \oplus r$的符号位，完成大小判断：若$d=1$，则$\alpha_i < \beta_i$；反之则$\alpha_i > \beta_i$。

#### 步骤3：检索结果判定

A方通过逐轮安全比较，完成二叉树全路径遍历：

- 若最终未匹配到对应叶子节点，判定目标$id$不存在于税务方数据库，协议终止；
- 若匹配到对应叶子节点，确认目标$id$存在，获取该$id$的完整索引值，进入下一阶段。

### 3.2 阶段二：基于同态加密的解密密钥安全协商

本阶段核心目标：在不泄露$MSK$与检索$id$的前提下，双方安全协商出分片解密密钥$K_{id}$，实现A方对匹配数据的解密。

#### 步骤1：税务方主密钥同态加密

B方使用加法同态加密算法的公钥$pk$，对系统主密钥$MSK$执行同态加密，得到密文$C_{MSK}=Enc_{HE}(MSK,pk)$，将$C_{MSK}$发送至A方。

#### 步骤2：公安方同态运算与盲化

A方生成大整数随机数$R$，基于加法同态加密的同态性质，对密文执行以下运算：

1. 同态加法：$C_{MSK+id} = C_{MSK} + Enc_{HE}(id,pk)$
2. 同态数乘：$C_{(MSK+id)\cdot R} = R \cdot C_{MSK+id}$
   得到盲化后的密文$C_{(MSK+id)\cdot R}$，将该密文发送回B方。

#### 步骤3：税务方解密与密钥材料生成

B方使用同态加密的私钥$sk$，对密文$C_{(MSK+id)\cdot R}$执行解密，得到明文$(MSK+id)\cdot R$；
基于该明文计算密钥材料$h = g^{\frac{1}{(MSK+id)\cdot R}}$，将$h$发送至A方。

#### 步骤4：公安方解盲与数据解密

A方对收到的密钥材料$h$执行幂运算，去除盲化因子$R$，得到解密密钥：

$$
h^R = \left( g^{\frac{1}{(MSK+id)\cdot R}} \right)^R = g^{\frac{1}{MSK+id}} = K_{id}
$$

A方使用$K_{id}$解密对应匹配的加密业务数据，完成整个协议流程。

## 四、安全性分析

### 4.1 核心隐私性保障

1. **检索索引隐私性**：公安方的检索$id$全程以盲化形式或密文形式交互，税务方仅能收到盲化后的$e_A$与同态密文，由于随机数$r,R$的密码学安全性，税务方无法反推出任何关于检索$id$的明文信息。
2. **全量数据隐私性**：税务方的全量索引$\beta_i$与主密钥$MSK$全程不泄露明文，公安方仅能通过协议获取大小关系与最终的解密密钥，无法反推出$\beta_i$与$MSK$的任何信息，也无法获取非匹配数据的任何内容。
3. **密码学原语安全性**：协议基于语义安全的同态加密算法、离散对数困难问题、安全数值比较协议构建，所有密码学组件均满足现代密码学安全要求，无明文隐私泄露风险。

### 4.2 正确性保障

1. 若检索$id$存在，协议可通过安全比较完成二叉树的正确匹配，最终可正确协商出解密密钥$K_{id}$，完成业务数据的解密；
2. 若检索$id$不存在，协议可正确判定无匹配节点，公安方无法获取任何有效密钥材料与数据信息。

## 五、算法性能

1. **离线预处理优化**：排序、分桶、二叉树构建、数据加密均为离线执行，可提前完成，不影响在线查询时延；10亿条数据的分桶二叉树可将检索轮次控制在30轮左右（$\log_2(10^9)\approx30$），大幅降低在线交互开销。
2. **国密算法适配**：政务场景推荐使用国密算法体系，同态加密推荐适配国密标准的BFV/BGV算法，对称加密使用SM4算法，哈希算法使用SM3算法，椭圆曲线群使用SM2曲线，满足国家密码应用合规要求。
3. **工程化优化**：可通过并行计算完成多桶的预处理与检索，支持批量检索场景；同态加密运算可通过硬件加速方案优化，降低在线计算时延。
